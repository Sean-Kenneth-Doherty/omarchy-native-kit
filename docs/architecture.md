# Architecture

## Principle

Omarchy owns themes. Apps consume them.

This kit should never pretend to be the theme source of truth. It adapts Omarchy's current theme into formats app authors can use.

## Data flow

```text
~/.config/omarchy/current/theme/colors.toml
        │
        ▼
  Omarchy theme reader
        │
        ▼
  Raw palette tokens
        │
        ▼
  Semantic token mapper
        │
        ├── CSS variables: :root { --omarchy-bg: ... }
        ├── JSON: { semantic: { background: ... } }
        ├── shell: export OMARCHY_BG=...
        └── future: GTK CSS / Qt palette / Tailwind preset
```

## Token model v0

### Raw palette

Mirrors current `colors.toml`:

```ts
type OmarchyRawPalette = {
  accent: string
  cursor: string
  foreground: string
  background: string
  selection_foreground: string
  selection_background: string
  color0: string
  // ... color15
}
```

### Semantic palette

```ts
type OmarchySemanticPalette = {
  background: string
  foreground: string
  surface: string
  surfaceRaised: string
  border: string
  borderStrong: string
  muted: string
  mutedForeground: string
  accent: string
  accentForeground: string
  focusRing: string
  selection: string
  selectionForeground: string
  danger: string
  success: string
  warning: string
  info: string
}
```

### Initial mapping

- `background` ← `background`
- `foreground` ← `foreground`
- `surface` ← blend(`background`, `foreground`, 0.06)
- `surfaceRaised` ← blend(`background`, `foreground`, 0.10)
- `border` ← blend(`background`, `foreground`, 0.18)
- `borderStrong` ← blend(`background`, `foreground`, 0.32)
- `muted` ← `color8`
- `mutedForeground` ← blend(`foreground`, `background`, 0.30)
- `accent` ← `accent` or `color4`
- `accentForeground` ← choose black/white-ish based on contrast
- `danger` ← `color1`
- `success` ← `color2`
- `warning` ← `color3` or `color11`
- `info` ← `color6` or `color4`

## Hook strategy

Omarchy has a single user hook path:

```text
~/.config/omarchy/hooks/theme-set
```

Problem: multiple apps may want to react to `theme-set`. Clobbering that file is rude.

Proposed convention:

```text
~/.config/omarchy/apps/<app-id>/theme-set
```

The kit can install a small dispatcher into `~/.config/omarchy/hooks/theme-set` only if safe:

```bash
#!/usr/bin/env bash
set -euo pipefail
for hook in "$HOME"/.config/omarchy/apps/*/theme-set; do
  [[ -x "$hook" ]] && "$hook" "$@"
done
```

If a hook already exists, CLI should ask user to merge or print manual instructions. Never silently overwrite.

## Files this project may write on user machines

Safe:

- project source files
- generated app files inside the app repo
- `~/.config/omarchy/apps/<app-id>/...`
- `~/.local/share/applications/<app-id>.desktop` when user runs install command

Unsafe by default:

- `~/.local/share/omarchy/*` — upstream-managed; read only
- existing `~/.config/omarchy/hooks/theme-set` unless user explicitly permits merge

## Template stance

The first template should be web tech because CSS variables are universal and quick. It can later wrap with Tauri for native windowing.

A good template includes:

- `src/theme.css` generated by CLI
- app shell using semantic variables
- command palette pattern
- keyboard shortcut examples
- `npm run theme:sync`
- `npm run doctor`
